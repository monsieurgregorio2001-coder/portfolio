Technical Build Plan: AI-Powered Recruitment CRM

UniversalHire - 120-Day Build Strategy

Executive Summary

This document outlines a comprehensive technical strategy to build an AI-first recruitment CRM from the ground up. The platform will unify ATS, sourcing, outreach, AI automation, and reporting into a modern, scalable system.

Timeline: 120 days (4 months)

  • Days 1-60: MVP with core CRM + AI v1
  • Days 61-120: Enterprise hardening + Advanced AI features

Team Structure:

  • UI/UX Designer: Week 4-17 (Days 22-119)
  • Full-Stack Developer #1: Week 4-17 (Days 22-119)

  • Framework: Next.js 14+ (App Router)
  • UI Library: Tailwind CSS + shadcn/ui
  • State Management: Zustand + React Query
  • Forms: React Hook Form + Zod
  • Drag & Drop: dnd-kit

Backend

  • Runtime: Node.js 20+ with TypeScript
  • Framework: Next.js API Routes (MVP) → NestJS (Post-MVP)
  • API Style: tRPC for type-safe APIs
  • Validation: Zod

Database

  • Primary Database: PostgreSQL 16 (Neon or Supabase)
  • Vector Database: pgvector (for semantic search)
  • Cache: Redis (Upstash)
  • ORM: Prisma

AI/ML Stack

  • LLM Provider: OpenAI (GPT-4o, GPT-4o-mini)
  • Embeddings: OpenAI text-embedding-3-small
  • Transcription: OpenAI Whisper API
  • AI Framework: Vercel AI SDK
  • Prompt Management: Langfuse

Infrastructure

  • Hosting: Vercel (frontend + API routes)
  • Object Storage: Cloudflare R2 or AWS S3

  • SMS: Twilio

  • Logs: Axiom or Betterstack

DevOps

  • Version Control: Git + GitHub
  • Monorepo Tool: Turborepo + pnpm
  • CI/CD: GitHub Actions
  • Authentication: Clerk
  • Background Jobs: Inngest or Trigger.dev
  • Feature Flags: GrowthBook
  • Testing: Vitest + Playwright

Multi-Tenancy Architecture

Core Principle: Every organization is completely isolated at the database level.

Data Isolation Strategy

Database Level:

  • Every table contains organization_id column
  • Row-Level Security (RLS) enforces data access
  • Users can belong to multiple organizations
  • Unique constraints scoped per organization
  • Cascade deletes for GDPR compliance

Application Level:

  • Middleware validates user membership in organization
  • All queries automatically filtered by organization context
  • Organization switcher in UI
  • Org-scoped routes: /[orgSlug]/contacts

Caching & Jobs:

  • Cache keys prefixed with organization ID
  • Background jobs carry organization context
  • Rate limiting per organization

Security Layers (Defense in Depth)

Layer 1: Frontend
  ↓ Organization switcher, context validation
Layer 2: API Middleware
  ↓ Validates user belongs to organization
Layer 3: Application Logic
  ↓ Explicitly filters by organization_id
Layer 4: Database RLS
  ↓ Final enforcement at PostgreSQL level
Layer 5: Audit Logs
  └→ Track all actions with org context

MULTI-TENANT DATA FLOW:
1. User logs in → JWT contains user_id
2. User selects/switches organization → org_id stored in session
3. Every API request includes org_id in context

5. All queries filtered by org_id (enforced by RLS at DB level)
6. Background jobs process data per-organization in isolation

Key Modular Principles

1. Domain-Driven Design

  • Each module owns its domain logic, data models, and business rules
  • Clear boundaries between modules
  • Modules communicate via well-defined interfaces

2. Monorepo Structure

  • Turborepo + pnpm for fast builds
  • Separate packages for each domain (contacts, companies, jobs)
  • Shared packages (database, auth, types, UI components)

3. Service Layer Pattern

  • Each module exposes clear service interfaces
  • Dependency injection for testability
  • Event-driven communication between modules

4. Benefits

  • Independent development by team members
  • Each module can be unit tested in isolation
  • Can extract modules into microservices later if needed
  • Clear boundaries reduce cognitive load

8. Key Technical Decisions & Trade-offs

Decision 1: Multi-Tenancy from Day 1

Choice: Build multi-tenant architecture from the foundation

Rationale:

  • Retrofitting multi-tenancy is extremely difficult
  • RLS provides database-level security
  • Critical for B2B SaaS business model

Trade-offs: Slightly more complex initial setup, but saves months of refactoring later

Decision 2: Modular Monolith vs. Microservices

Choice: Start with modular monolith, extract services later if needed

Rationale:

  • Faster iteration, simpler deployment
  • Clear module boundaries allow future extraction
  • Can always split later when scale demands it

Trade-offs: All modules share same deployment, but speed matters more than scale in early days

Decision 3: Build vs. Buy (Calendar/Email)

Choice: Integrate existing APIs (Gmail, Outlook, Google Calendar)

Rationale: Focus engineering time on unique AI value, not commodity features

Trade-offs: Dependent on third-party reliability, but worth it

Decision 4: Team Size (4 People Max)

Choice: Keep team small and focused

Rationale:

  • Less coordination overhead
  • Faster decision-making
  • Higher individual ownership
  • Modern tools enable small teams to move fast

Trade-offs: Longer timeline for some features, but higher quality output

Conclusion

  1. Strong technical foundation (Weeks 1-4) - don’t rush this
  2. Clear team roles - Everyone knows their domain
  3. AI as enhancement - Ship CRM functionality first, layer AI
  4. Weekly demos - Fast feedback and decision-making
